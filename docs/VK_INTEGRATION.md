# Интеграция с ВКонтакте: оплата, вывод голосов и общие требования

Документ описывает требования ВК к мини-приложению: оплата голосами, вывод средств и общая интеграция. Реализация в проекте: `lib/vk-bridge.ts`, `lib/vk-oauth.ts`, экраны магазина и вывода.

---

## 0. Вход через ВК на своём сервере

Приложение можно открывать **на своём домене** (не только в мини-приложении ВК). В этом случае вход выполняется через **VK OAuth** (Implicit Flow).

- **Настройка:** в корне проекта создайте `.env` (см. `.env.example`) и укажите `NEXT_PUBLIC_VK_APP_ID` — ID приложения из [Управления приложениями ВК](https://vk.com/apps?act=manage).
- В настройках приложения ВК нужно указать **Redirect URI** (адрес возврата после входа). **Где именно это сделать** — пошагово описано в **[docs/VK_REDIRECT_URI_НАСТРОЙКА.md](VK_REDIRECT_URI_НАСТРОЙКА.md)**: раздел «Платформы» → платформа «Веб»/«Сайт» → поле «Адрес сайта» или «Доверенный redirect URI». Укажите адрес главной страницы приложения, например `http://localhost:3000/` или `https://ваш-домен.com/`. Значение должно **посимвольно** совпадать с тем, что уходит в OAuth. При ошибке **«Security Error»** задайте в `.env` переменную `NEXT_PUBLIC_VK_REDIRECT_URI` с тем же URL, что и в настройках ВК.
- **Логика:** при нажатии «Войти через ВКонтакте» выполняется **переход** на страницу авторизации ВК. После согласия пользователя ВК возвращает на ваш сайт (на ту же страницу, с токеном в hash). Сессия сохраняется в `localStorage` и восстанавливается после перезагрузки.
- **Оплата и вывод** на своём сервере по-прежнему привязаны к ВК (в мини-приложении доступны голоса и вывод). Позже можно подключить сторонние платёжные системы.

Реализация: `lib/vk-oauth.ts`, вызов в `lib/game-context.tsx` (логин, восстановление сессии, выход).

## 1. Общие требования к VK Mini Apps

- **VKWebAppInit** — вызывать при загрузке приложения (уже в `initVKBridge()`).
- **Скрипт приложения** — должен открываться в iframe ВК (или с параметрами запуска ВК).
- **Библиотека** — использовать официальный [VK Bridge](https://dev.vk.com/bridge) (`@vkontakte/vk-bridge`).
- **Правила платформы** — соответствовать [правилам размещения Mini Apps](https://dev.vk.com/mini-apps-rules) (контент, платежи, персональные данные).
- **Домен** — приложение должно быть размещено на домене, указанном в настройках приложения ВК.
- **Данные пользователя** — получать только через методы Bridge (например, `VKWebAppGetUserInfo`), не хранить пароли ВК.

## 2. Оплата голосами (пополнение баланса)

### 2.1 Метод ВК

- **VKWebAppOpenPayForm** — открытие формы оплаты (голоса или VK Pay).
- Параметры (актуально уточнять на [dev.vk.com](https://dev.vk.com)):
  - **action** — тип операции, например `pay-to-service` (платёж в сервис).
  - **amount** — сумма в голосах (целое число).
  - **app_id** — ID приложения ВК (из настроек приложения).
  - **hash** (или **sign** / **merchant_sign**) — подпись запроса, формируется на **бэкенде** из параметров и секретного ключа (без подписи оплата может не пройти модерацию/проверку).

### 2.2 Требования ВК к платежам

- Не зачислять виртуальные ценности до **подтверждения платежа** (результат от ВК или callback на сервер).
- Использовать **серверную верификацию**: после успешного ответа формы ВК отправлять на свой бэкенд запрос с `order_id`/параметрами платежа; бэкенд проверяет платёж через API ВК и только после этого начисляет голоса в БД.
- Не хранить платёжные реквизиты и секретные ключи на клиенте; **app_id** можно передавать с клиента, **hash/sign** — только с бэкенда.
- Отображать пользователю понятное описание товара/услуги и сумму до открытия формы оплаты.
- Обрабатывать отмену платежа пользователем и ошибки (не показывать «успех» при отмене).

### 2.3 Реализация в проекте

- **Магазин** (`components/shop-screen.tsx`): кнопки «10/50/100/500 голосов» вызывают `purchaseVKVoices(amount)`.
- **vk-bridge** (`lib/vk-bridge.ts`): `purchaseVKVoices(amount)` вызывает `VKWebAppOpenPayForm` с `action: "pay-to-service"`, `amount`. В продакшене нужно:
  - запрашивать с бэкенда `app_id` и `hash` (или аналог) для выбранной суммы;
  - передавать их в `VKWebAppOpenPayForm`;
  - по результату вызова (или по callback с сервера) обновлять баланс только после подтверждения на бэкенде.

## 3. Вывод голосов

### 3.1 Требования ВК и практика

- Вывод средств (перевод голосов пользователю) обычно **не** делается одним только методом Bridge в клиенте. Используется:
  - **бэкенд приложения**: пользователь создаёт заявку на вывод в мини-приложении, бэкенд получает заявку, проверяет баланс и лимиты, затем выполняет перевод через API ВК или ручную обработку;
  - либо интеграция с партнёрскими платёжными решениями по документации ВК.

### 3.2 Ограничения и правила в приложении

- **Минимальная сумма вывода** — в приложении: 10 голосов (и баланс не менее 200 для доступа к экрану вывода).
- **Максимум в день** — в приложении: 10 000 голосов (лимит за сутки, сброс по дате).
- **Списание баланса** — списывать только после того, как бэкенд подтвердил приём заявки (или выполнил перевод). Иначе при сбое пользователь теряет голоса без вывода.
- Указывать пользователю срок обработки (например: «Обработка до 24 часов») и не гарантировать мгновенный вывод без интеграции с платёжной системой.

### 3.3 Реализация в проекте

- **Экран вывода** (`components/withdraw-screen.tsx`): ввод суммы, проверка лимитов (мин. 10, макс. остаток за день 10 000, баланс ≥ 200), вызов `requestWithdraw(amount)`.
- **vk-bridge** (`lib/vk-bridge.ts`): `requestWithdraw(amount)` сейчас симулирует успех. В продакшене должен:
  - отправлять на бэкенд запрос на вывод (user_id, amount);
  - бэкенд создаёт заявку, проверяет лимиты и баланс;
  - после принятия заявки бэкенд возвращает успех, тогда клиент списывает баланс и вызывает `recordWithdraw(amount)`.

## 4. Пользователь и данные

- **VKWebAppGetUserInfo** — используется для отображения имени и аватара (id, first_name, last_name, photo_100, photo_200).
- Не запрашивать лишние права; хранить только необходимые данные в соответствии с политикой конфиденциальности и правилами ВК.
- Опция «Скрыть аватар ВК» (за 100 голосов) — локальное предпочтение пользователя, не передавать фото в другие сервисы без согласия.

## 6. Хранимые процедуры (VKScript)

Хранимые процедуры — функции на языке **VKScript**, код которых хранится в настройках мини-приложения (**Разработка → Хранимые процедуры**), а выполняется на серверах ВКонтакте. Это позволяет выполнять несколько API-запросов без передачи промежуточных результатов на устройство (быстрее и меньше трафика). Сервер мини-приложения для их работы не обязателен.

### 6.1 Настройка в ВК

1. [Управление приложениями](https://vk.com/apps?act=manage) → настройки приложения → **Разработка → Хранимые процедуры**.
2. **Добавить процедуру** — имя латиницей (1–100 символов). При вызове всегда используется префикс `execute.` (например, процедура `getUserBalance` вызывается как `execute.getUserBalance`).
3. Указать **версию API ВКонтакте** (например 5.134).
4. Написать код на VKScript: параметры — `Args.param_name`, вызов API — `API.methodName(...)`, результат запроса — `r@.field_name`, возврат — `return { "result": ... };`.
5. При необходимости создавать **версии процедуры** (разные версии кода для совместимости со старым клиентом).

Подробно: [Хранимые процедуры — dev.vk.com](https://dev.vk.com/ru/mini-apps/settings/development/stored-procedures).

### 6.2 Вызов из приложения

В проекте реализован вызов через **VKWebAppCallAPIMethod** в модуле `lib/vk-stored-procedures.ts`:

- **callStoredProcedure\<T\>(procedureName, params, options?)** — вызывает `execute.ИМЯ_ПРОЦЕДУРЫ` с указанными параметрами.
- **VK_API_VERSION** — константа версии API (по умолчанию 5.134).
- Параметры `params` в коде процедуры доступны как `Args.param_name`. Опционально: `apiVersion`, `funcVersion` (версия процедуры при нескольких версиях).

Пример: получение данных одним запросом с сервера ВК без своего бэкенда — создать процедуру в настройках приложения и вызывать её из кода через `callStoredProcedure("имяПроцедуры", { param1: value1 })`.
